# v4.9.0 实施计划 — 编排深化

> 基于 [产品策略](./product-strategy.md) 主线 B「编排能力深度」
> 创建：2026-07-23，目标版本：v4.9.0

## 目标

补齐编排能力的高价值缺口，让工作流从"能用"升级为"好用"，并收敛文档与安全债。

## 任务清单

### P0-1：Workflow 持久化 + 执行历史

**问题**：当前 `WorkflowEngine.execute()` 执行完即丢，App 重启后无任何历史记录，用户无法回看/复用。

**现状**：
- Android `WorkflowExecutionState` 是纯内存 StateFlow，无落库
- AppDatabase v9 仅 6 实体（Session/Message/AgentConfig/ActivityLog/Plugin/Task），无 Workflow 相关表
- iOS SwiftData `AppSchemaV1` 同样无 Workflow 模型
- 协议层 `workflow-schema.json` 无历史记录契约

**实施步骤**：

1. **协议层先行** — `protocol/schemas/workflow-schema.json` 新增 `WorkflowRunRecord` 定义
   - 字段：`id / workflowId / workflowName / input / output / startedAt / completedAt / status(Running|Completed|Failed|Cancelled) / failedNodeIds / error / logs`
   - 作为双端实现的统一契约

2. **Android 数据层** — AppDatabase v9→v10
   - 新增 `WorkflowRunEntity`（字段对齐 `WorkflowRunRecord`，`failedNodeIds`/`logs` 序列化为 JSON，复用 `Converters.kt`）
   - 新增 `WorkflowRunDao`：`insert / updateStatus / getRecentFlow(limit) / getByWorkflowIdFlow / deleteOlderThan(timestamp)`
   - `MIGRATION_9_10`：建 `workflow_runs` 表 + 索引（`workflowId`、`startedAt DESC`）
   - `AppDatabase` 版本号升 10，注册 migration

3. **iOS 数据层** — SwiftData `AppSchemaV1`→`AppSchemaV2`
   - 新增 `WorkflowRunEntity` 模型，字段对齐
   - `MigrationPlan` 增加 `V1→V2` lightweight stage
   - `DataController` 暴露 `fetchRecentRuns / saveRun / deleteOldRuns` 方法

4. **WorkflowEngine 改造**（双端）
   - 注入 `WorkflowRunDao` / `DataController`
   - `execute()` 入口 `insert(running_record)`
   - try/catch/finally 中 `updateStatus(completed/failed)` + 写入 output/error/logs
   - 新增 `getHistoryFlow(workflowId)` 供 UI 订阅

5. **UI** — 执行历史列表
   - Android `WorkflowScreen` 新增 "历史" Tab，按 `startedAt DESC` 展示
   - 点击历史项展开节点级输出/失败/日志回放
   - iOS `WorkflowView` 同步

**验收**：
- 执行工作流后，重启 App 仍可看到历史记录
- 历史列表按时间倒序，展示状态/耗时/输入输出预览
- 单元测试：`WorkflowRunDao` CRUD + `WorkflowEngine` 写历史

### P0-2：填平 CollaborationManager stub（双端，本地 MVP）

**问题**：Android/iOS CollaborationManager 均为 stub（Android 用 `delay` 模拟网络，iOS 6 处 TODO），双端功能不对齐。

**现状**：
- iOS 已有本地 `sessions.json` 缓存 + 数据模型（Participant/CollabSession/ParticipantRole）
- Android 同样是 stub，且更简陋
- 协议层无协作信令 schema

**实施步骤**（本地 MVP，不做真实信令）：

1. **收敛 stub 注释** — 双端 6 处 TODO 改为明确状态
   - 注释格式：`// MVP: 本地会话状态已实现，真实信令推迟到 v5.2.0（见 docs/product-strategy.md）`
   - 避免 stub 长期悬空误导

2. **Android 本地 MVP 补齐** — 对齐 iOS 已有能力
   - `CollaborationManager.kt` 移除 `delay` 模拟，改为真实本地状态管理
   - 复用 `sessions.json` 模式持久化（DataStore 或 Room）
   - 实现：createSession（本地生成 sessionId+code）/ joinSession（本地写入）/ leaveSession（本地移除）

3. **UI 反馈** — 双端会话状态可视化
   - Host 生成邀请码后展示二维码（复用现有 QR 生成能力，如有）
   - 加入方扫码后 UI 反馈"已加入会话"
   - 明确标注"本地会话，不跨设备同步"

4. **协议层草案** — `protocol/schemas/collab-schema.json`（草案，标注 `status: draft`）
   - 定义信令帧：`session:created / join / leave / participant:joined / participant:left`
   - 作为 v5.2.0 真实信令的契约预留，但 v4.9.0 不实现传输层

**验收**：
- 无 `TODO` 字样的悬空 stub（全部收敛为"MVP 已实现"或"推迟到 vX"明确注释）
- Android 端能力与 iOS 对齐
- 协议层有 collab-schema 草案

### P1-1：证书锁定填真实 pin

**问题**：`CertificatePinnerFactory` pin map 为空，框架已建但形同虚设。iOS 三处 TLS TODO 引用不存在的文档。

**现状**：
- Android `PUBLIC_API_PINS` 全部注释，`TransportFactory` 硬编码 `enableCertificatePinning = false`
- iOS `OpenAIHTTPTransport / WebSocketTransport / McClient` 三处 TODO
- `protocol/transport/tls-pinning.md` 不存在（被引用但缺失）

**实施步骤**：

1. **补文档** — `protocol/transport/tls-pinning.md`
   - 说明 pin 策略、应锁定域名、SPKI 获取方法、轮换流程
   - 作为双端 pin 列表的单一事实来源

2. **获取 SPKI SHA-256**（每个域名 primary + backup）
   - `api.openai.com`（OpenAI 兼容 AgentType 共用）
   - `generativelanguage.googleapis.com`（Gemini，候选）
   - 可选：`openclaw.supplies` / `clawhub.ai`（Marketplace 公网请求）
   - **不锁定**：所有 `127.0.0.1` / `localhost` / 用户自定义 serverUrl

3. **Android 填值 + 启用**
   - `CertificatePinnerFactory.PUBLIC_API_PINS` 填入 primary+backup pin
   - `TransportFactory` 改为 `enableCertificatePinning = isPublicEndpoint(config.serverUrl)`（动态判断）
   - 保留用户设置开关（Settings → 安全 → 证书锁定），默认开启

4. **iOS 实现 URLSessionDelegate**
   - 新建 `TLSPinningDelegate`：`urlSession(_:didReceive:completionHandler:)` 校验叶子证书 SPKI
   - 注入 `OpenAIHTTPTransport.init()` / `WebSocketTransport` / `McpClient` 的 `URLSession(configuration:delegate:)`
   - pin 列表与 Android 一致（从 `protocol/transport/tls-pinning.md` 同步）

**验收**：
- `api.openai.com` 连接时校验 SPKI，pin 不匹配则连接失败
- 本地端点（127.0.0.1）不受影响
- iOS 三处 TODO 消除，引用的文档存在

**风险**：服务端密钥轮换会导致 pin 失效。必须配 backup pin，建议建立轮换监控（后续 P2）。

### P1-2：Marketplace 收藏

**问题**：Marketplace 仅浏览，无收藏/订阅机制，用户无法标记感兴趣的 Agent 以便后续快速找到。

**现状**：
- 无 favorites/bookmark 数据模型、DAO、UI
- `installedIds` 仅内存运行时状态，不持久化
- iOS `MarketplaceAgent.swift` 字段更丰富但同样无收藏

**实施步骤**：

1. **数据模型** — `MarketplaceFavoriteEntity`（双端对齐）
   - 字段：`agentId(主键) / name / type / serverUrl / addedAt`
   - 冗余 name/type/serverUrl 便于从收藏直接安装，无需重新查询 API

2. **Android 持久层** — AppDatabase v10→v11（若与 P0-1 同期则合并到同一 Migration 链）
   - 新增 `MarketplaceFavoriteDao`：`upsert / delete / getAllFlow / isFavorite(agentId)`
   - `MIGRATION_10_11`（或与 9→10 合并）：建 `marketplace_favorites` 表

3. **iOS 持久层** — SwiftData `AppSchemaV2`（若 P0-1 已升 V2 则合并）
   - 新增 `MarketplaceFavoriteEntity` 模型
   - `DataController` 暴露收藏 CRUD

4. **UI 改造**（双端）
   - MarketplaceAgentCard 右上角加书签图标（`Icons.Default.Bookmark` / `bookmark` / `bookmark.fill`）
   - 点击 toggle 收藏，实时刷新（Flow/@Observable 订阅）
   - 顶部加"收藏"FilterChip，筛选仅展示已收藏
   - 收藏列表项支持直接"安装"（复用现有 install 流程）

**验收**：
- 点击书签可收藏/取消收藏，重启 App 保留
- "收藏"筛选可过滤已收藏 Agent
- 从收藏可直接安装
